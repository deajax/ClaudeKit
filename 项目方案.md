# Claude CLI Desktop — 跨平台桌面管理工具

## 项目概述

### 项目目标

构建一个跨平台（macOS / Windows）桌面工具，围绕 Claude Code CLI 提供以下能力：

- **快速切换**：在不同模型商、不同 API 配置以及不同功能之间快速切换
- **终端集成**：内置稳定终端，支持持久化会话
- **环境管理**：统一管理 shell 环境变量，支持自动写入配置文件
- **配置免登录**：一键完成 Claude Code 免登录配置

### 目标用户

经常使用 Claude Code CLI 的开发者，需要在多个模型/API 配置之间切换。

## 配置方案分析（重要）

### Claude Code 的配置机制

Claude Code 通过环境变量控制模型提供商、API 密钥等行为，支持两种配置方式：

**方式一：Shell 环境变量**
```bash
export ANTHROPIC_BASE_URL=https://api.deepseek.com/anthropic
export ANTHROPIC_AUTH_TOKEN=sk-xxxx
export ANTHROPIC_MODEL=deepseek-v4-pro
```
终端启动 `claude` 时自动读取。

**方式二：settings.json 的 `env` 字段**
```json
{
  "env": {
    "ANTHROPIC_BASE_URL": "https://api.deepseek.com/anthropic",
    "ANTHROPIC_AUTH_TOKEN": "sk-xxxx"
  }
}
```
作用于 `~/.claude/settings.json`（用户级）或 `.claude/settings.local.json`（项目级），每个会话生效。

### 优先级关系

settings.json 的 `env` 字段 **优先级高于** shell 环境变量。如果两者都配置了同一变量，settings.json 会覆盖 shell 中的值，甚至可能因配置冲突导致报错。

### 推荐方案：使用环境变量注入

本工具采用 **环境变量注入** 作为主要配置方式，理由如下：

1. **更可控**：桌面工具通过 node-pty 创建终端时可以精确注入所需环境变量，不污染系统级配置
2. **官方推荐**：大多数模型商（DeepSeek、OpenRouter、阿里云百炼）的教程都使用环境变量方式
3. **避免冲突**：不修改 Claude 自身的 settings.json，避免因配置冲突导致奇怪的报错
4. **职责分离**：工具管理 shell 配置，Claude 管理自身设置，各司其职
5. **即时生效**：切换 Profile 时只需重启终端进程，无需修改配置文件再重载

**但以下场景例外，需要使用 settings.json/settings.local.json 方式：**
- 一键免登录配置（修改 `~/.claude.json`）
- 用户手动选择"写入配置文件"进行持久化

### 关键环境变量

| 变量 | 用途 | 必填 |
|------|------|------|
| `ANTHROPIC_BASE_URL` | 模型商的 API 端点 | ✅ |
| `ANTHROPIC_AUTH_TOKEN` | API 认证令牌 | ✅ |
| `ANTHROPIC_MODEL` | 默认使用的模型 | ❌（推荐配置） |
| `ANTHROPIC_DEFAULT_OPUS_MODEL` | 覆盖旗舰模型 | ❌ |
| `ANTHROPIC_DEFAULT_SONNET_MODEL` | 覆盖高级模型 | ❌ |
| `ANTHROPIC_DEFAULT_HAIKU_MODEL` | 覆盖初级模型 | ❌ |

最多可同时配置 4 个模型（1 个默认 + 3 个分级覆盖）。

## 「JSON 数据库」设计

### 核心概念（项目铁律）

项目不使用传统数据库，而是用 **JSON 文件** 作为持久化方案，这是贯穿整个项目的核心规则：

- **每个功能模块都必须有自己的 JSON 数据库文件**，哪怕是空的 `{}`
- 以后每新增一个功能，就对应新增一个 JSON 文件
- 项目模式固定为：**Vue + TypeScript + JSON 数据库**
- **双向修改**：支持读取 / 写入 / 删除
- **统一存储路径**：`~/.ClaudeCLI/` 目录下
- **内置数据**：预置模型商、快速运行任务等

### JSON 数据库文件清单

| 文件名 | 用途 | 主要字段 |
|--------|------|----------|
| `providers.json` | 模型商管理 | `name`, `icon`(base64), `BASE_URL`, `AUTH_TOKEN`, `models[]`, `model`(默认), `opusModel`, `sonnetModel`, `haikuModel`, `thinkingSupported`, `balanceApi` |
| `profiles.json` | 快速切换配置集 | `name`, `providerId`, `model`, `envVars`, `active` |
| `env.json` | 环境变量管理 | `key`, `value`, `source`, `autoWrite` |
| `tasks.json` | 快速运行任务 | `name`, `command`, `cwd`, `providerId` |
| `terminal.json` | 终端标签页状态 | `tabs[]` (`id`, `title`, `cwd`, `createdAt`), `activeTabId` |
| `settings.json` | 应用设置 | `theme`(dark/light), `scrollback`(回滚缓冲行数), `shell`, `fontSize`, `fontFamily`, `loginMode` |

**说明**：
- `terminal.json`：记录所有标签页的标题、路径等信息。终端进程关闭后内容不保留，但标签页命名持续存在。只有用户主动关闭选项卡，才从 JSON 中删除该条目。
- `settings.json`：主题支持 dark / light 切换；`scrollback` 控制终端回滚缓冲的行数。

### 数据操作接口（IPC）

所有 JSON 数据库操作通过 IPC 通道进行：

```json
db:read      -> 读取指定 JSON 文件
db:write     -> 写入指定 JSON 文件
db:delete    -> 删除指定 JSON 文件中的条目
db:list      -> 列出指定 JSON 文件所有条目
```

## 核心功能

### 0. UI 布局

整个应用以终端为主体，摒弃传统多页面结构，采用单页布局。不使用 Electron 默认工具栏，完全自定义窗口外观。

**窗口与标题栏**：
- 使用 Electron 无边框窗口（`frame: false`），去掉默认工具栏
- 顶部第一行就是终端标签栏
- macOS：标签栏左方留出空间避开交通灯（红绿灯），内容不与其重叠
- Windows：标签栏右方留出空间避开窗口控件按钮（最小化/最大化/关闭）
- 拖拽区域：标签栏空白区域可拖拽移动窗口

```

│ Tab 1│ Tab 2│  [+ New Tab]              │

└─────────────────────────────────────────┘

弹窗/抽屉（打开覆盖在终端上方）：

  - 设置（Drawer）

  - 帮助文档（Drawer）
  - 关于（Modal）
```

### 1. 终端管理（主体）

- 基于 xterm.js + node-pty 的完整终端模拟
- **多标签页**：支持同时打开多个终端标签页，每个独立会话
- 标签页可关闭、重命名
- 持久化会话历史记录
- 标签页支持拖拽排序
- 环境检测：启动终端前检查 Claude Code 是否已安装
  - 未安装则提供一键安装功能

**Claude 启动方式**：
- 终端不自启动 Claude Code
- 每个标签页内提供 **▶ Claude** 按钮，用户点击后在当前终端执行 `claude`
- Claude 运行中时 ▶ 按钮置灰，同时显示：
  - **🔄 重启**：停止当前 Claude 进程，重新执行启动命令
  - **⏹ 停止**：退出/停止 Claude（相当于终端内 `exit` 或 `Ctrl+C`）
- 用户也可在终端内直接 `exit` 或 `Ctrl+C` 退出 Claude，退出后按钮恢复，可再次启动

**标签页规则**（重要）：
- 至少存在 **1 个**选项卡，且至少选中 **1 个**（不固定是哪一个）
- 禁止关闭最后一个选项卡：当只剩一个标签页时，关闭操作被阻止并弹出 message 提醒
- 关闭选项卡 = 从 `terminal.json` 中删除该条目

### 2. 工具栏——模型/提供商快速切换

- 顶部工具栏下拉框，**一键切换当前使用的模型商和模型**
- 切换时通过 node-pty 注入环境变量（`ANTHROPIC_BASE_URL`、`ANTHROPIC_AUTH_TOKEN` 等）
- 点击「应用」后需要 **重启当前终端 + 重启 Claude Code** 才能生效
- 如果终端正在运行任务，需弹窗确认「任务将中断，是否重启？」
- 重启后自动在新的终端会话中启动 `claude`

### 3. 工具栏——运行任务

**UI 形式**：点击「运行任务」按钮 → 从顶部向下弹出 Drawer，类似 VS Code 的「终端 → 运行任务」

```
┌─────────────────────────────────────────┐
│  🔍 搜索命令或名称...                    │  ← 搜索栏，模糊匹配
├─────────────────────────────────────────┤
│  ○ npm run dev          [⚙]            │  ← 任务列表，右侧 ⚙ hover 显示
│  ○ npm run build        [⚙]            │
│  ○ npm run lint         [⚙]            │
│  ○ npm run test         [⚙]            │
│  ...                                    │  ← 超过最大高度滚动
├─────────────────────────────────────────┤
│  [+ 添加任务]                           │  ← 点击打开添加 form 弹窗
└─────────────────────────────────────────┘
```

**详细规则**：
- **搜索栏**：输入文字后实时模糊搜索任务名称和命令
- **任务列表**：设置最大高度，超过则滚动；每行右侧的配置按钮（⚙️）hover 时才显示，点击可编辑或删除
- **添加任务**：点击底部「+ 添加任务」按钮 → 弹出 Modal 表单（任务名称、执行命令、工作目录等）
- **执行**：点击任意任务 → 收起 Drawer → 在当前终端或新标签页执行命令
- **CRUD**：支持增删改查，数据存储在 `tasks.json`
- **内置任务**：预置 `npm run dev`、`npm run build`、`npm run lint`、`npm run test` 等
- 执行环境的模型商配置跟随当前选中配置

### 4. 更多菜单——模型商配置（Drawer → Modal）

**入口**：工具栏「更多」按钮 → 下拉菜单 → 模型商配置

**列表视图**（Drawer）：
- 以卡片或列表形式展示所有已配置的模型商
- 每个卡片显示：图标、名称、已配置模型数
- 支持新增、编辑、删除

**编辑视图**（Modal）：
- 模型商名称（必填）
- 模型商图标：支持上传图片，转成 base64 存入 `providers.json`
- Claude Code 配置项（对应环境变量）：
  - `ANTHROPIC_BASE_URL`（必填）
  - `ANTHROPIC_AUTH_TOKEN`（必填）
  - `ANTHROPIC_MODEL` — 默认模型（推荐）
  - `ANTHROPIC_DEFAULT_OPUS_MODEL` — 覆盖 Opus 模型（可选）
  - `ANTHROPIC_DEFAULT_SONNET_MODEL` — 覆盖 Sonnet 模型（可选）
  - `ANTHROPIC_DEFAULT_HAIKU_MODEL` — 覆盖 Haiku 模型（可选）
- 模型配置规则：
  - 最多配置 **4 个模型**（1 默认 + 3 分级覆盖）
  - 前三项（BASE_URL、AUTH_TOKEN、MODEL）是必填
  - OPUS / SONNET / HAIKU 是可选的
  - 必填项不填时不允许激活，需提示用户
- ⚠️ 注意：`BASE_URL` 使用的是 **Anthropic API 兼容接口**，不是所有模型商都支持

**内置模型商数据**（`~/.ClaudeCLI/providers.json` 预置）：

| 模型商 | BASE_URL | 说明 |
|--------|----------|------|
| 阿里云百炼 | `https://dashscope.aliyuncs.com/compatible-mode/v1` | 需配置 API Key |
| DeepSeek | `https://api.deepseek.com/anthropic` | 支持查询余额 |
| OpenRouter | `https://openrouter.ai/api` | 需额外设置 `ANTHROPIC_API_KEY=""` |
| 硅基流动 | `https://api.siliconflow.cn/v1` | Anthropic API 兼容 |

**特殊功能**：
- 支持查询余额的模型商在界面中提供「手动刷新查询余额」功能（放在底部状态栏显示）
- 查询余额接口不自动调用，用户手动触发，避免不必要的 token 消耗

### 5. 更多菜单——设置（Drawer）

**应用设置**：
- 主题切换（dark / light）
- 终端字体、字号
- 回滚缓冲行数（scrollback）
- Shell 类型（bash / zsh / powershell）

**Claude Code 免登录配置**（重要）：
- 本质：在 `~/.claude.json` 文件中设置 `"hasCompletedOnboarding": true`
- 操作逻辑：
  1. 读取 `~/.claude.json`
  2. 判断 `hasCompletedOnboarding` 字段：
     - 不存在 → 追加
     - 存在且为 `false` → 改为 `true`
     - 已经是 `true` → 不做任何修改
  3. 保留文件中所有其他原有内容不动
- 需要权限提升（sudo-prompt）写入文件，必须弹窗让用户确认，不可自提权
- 配置完成后需要 **重启终端 + 重启 Claude Code** 才能生效
- 此功能需要在帮助文档中重点说明

**其他功能**：
- 重置设置（恢复默认）
- 重置模型商（清空自定义模型商）
- 重置任务（清空自定义任务）
- 提 Issues 链接（跳转到 GitHub Issues）
- 仓库地址链接

**首次启动引导向导**（完整流程）：
- 第一次启动时自动弹出引导向导，包含以下步骤：

  **Step 1 — 环境检测**
  - 检测当前系统 Shell 类型（bash / zsh / powershell）
  - 检测 Shell 配置文件（`~/.zshrc` / `~/.bash_profile` / `~/.bashrc`）
  - 检测 Node.js 是否已安装及版本
  - 检测 npm 是否可用

  **Step 2 — 安装 Claude Code**
  - 检测本机是否已安装 Claude Code CLI
  - 如果未安装，提供一键安装（`npm install -g @anthropic-ai/claude-code`）
  - 安装完成后验证版本

  **Step 3 — 配置模型商**
  - 展示内置模型商列表（阿里云百炼、DeepSeek、OpenRouter、硅基流动）
  - 用户最少选择 1 个并填入 API Key 等必填信息
  - 支持跳过此步骤，之后再配置

  **Step 4 — 免登录配置**
  - 检查 `~/.claude.json` 中的 `hasCompletedOnboarding` 字段
  - 如未配置，自动写入 `"hasCompletedOnboarding": true`
  - 需要提权操作时请求用户授权

  **Step 5 — 完成**
  - 提示用户重启程序以使所有配置生效
  - 点击「完成」后进入主界面
  - 可选择「不再显示引导」

- 提供「跳过」按钮，可随时跳过向导进入主界面
- 之后可通过 设置 → 运行引导向导 重新触发

### 6. 更多菜单——环境变量文件管理（Drawer）

**设计原则**：尊重各平台原生 UI 风格，降低用户学习成本。

**macOS**（以文件编辑器形式）：
- 直接读取 `.zshrc` / `.bashrc` / `.bash_profile` 等 Shell 配置文件
- 显示为**代码编辑器**，支持：
  - 代码高亮（语法高亮 Shell 语法）
  - 代码格式化
  - 纯文本编辑
  - 保存按钮（写入文件）
- 权限说明：保存时如遇权限不足，弹窗请求用户确认（sudo-prompt）
- 后续预留：后期可能扩展支持 `.claude.json`、`settings.json`、`CLAUDE.md` 等文件的编辑（界面留入口）

**Windows**（以可视化表格形式）：
- 打开系统环境变量管理风格界面
- 显示为**可编辑表格**：
  - 变量名 / 变量值 两列
  - 双击单元格可编辑
  - 支持添加一行
  - 支持删除一行
- 底部「添加变量」按钮，点击弹 Form 或直接追加行
- 修改用户环境变量（非系统变量）

**通用**：
- 数据存储在 `env.json` 和 `profiles.json`
- 任何需要提权的写入操作，必须弹窗让用户确认，不可自提权

### 10. 版本检测与更新

**版本管理策略**：
- 以 GitHub Releases 为准
- 通过 GitHub API 检测最新 Release 版本号
- package.json 中的 `version` 字段为本地版本

**检测流程**（手动触发）：
1. 用户点击「检查更新」（放在 设置 或 关于 中）
2. 调用 `GET https://api.github.com/repos/{owner}/{repo}/releases/latest`
3. 获取最新 `tag_name` 与本地版本比较
4. 判断结果：
   - 有新版 → 弹窗提示：`v{旧版} → v{新版}`，显示更新内容（Release body），提供「前往下载」按钮跳转 GitHub Releases 页面
   - 已最新 → message 提示「当前已是最新版本」
5. 用户手动下载安装包覆盖安装，覆盖时保持 `~/.ClaudeCLI/` 数据不变，不重复触发引导向导

**.gitignore 配置**：
- `node_modules/`、`dist/`、`out/` 等构建产物
- `~/.ClaudeCLI/` 用户数据目录
- 确保用户配置不被提交或覆盖

**CI/CD**：
- 配置 GitHub Actions 自动化发布流程
- 打 tag 自动触发构建 + 生成 Release
